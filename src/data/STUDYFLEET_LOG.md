# 프로젝트 정리: studyfleet (스터디 매칭 및 협업 관리 플랫폼)

1️⃣ 전체 구조

① 프로젝트 한 줄 요약
**"스터디 모집부터 운영 전 과정을 트랜잭션 기반으로 관리하고, 도메인 간 정합성을 보장하는 스터디 협업 플랫폼"**

② 기획 배경 (문제 정의)
- **모집의 어려움**: 정보 파편화로 인한 원하는 조건의 스터디 검색 피로도.
- **운영의 번거로움**: 여러 도구(노션, 드라이브 등)를 번갈아 사용하는 운영 비효율성.
- **참여도 저하**: 명확한 목표와 진행 상태를 실시간으로 확인하기 어려워 중도 이탈 발생.

③ 해결 전략 (핵심 아이디어)
- **세분화된 권한 체계**: 리더, 멤버, 관리자 역할을 구분하여 스터디장 중심의 안정적인 운영 환경 제공.
- **데이터 무결성 보장**: 트랜잭션 기반의 상태 변화 관리를 통해 도메인 간 정합성 유지.

2️⃣ 기술 스택 & 인프라

- **Backend**: Spring Boot 3.5.7, Java 17
- **Database & Persistence**: MySQL 8.x, **MyBatis 3.0.5**
  - *선택 배경: 통계/조인 쿼리가 많은 도메인 특성상 SQL 직접 제어를 통해 최적화 및 가시성을 확보하고자 함.*
- **Frontend**: Vue 3.5.25, Vite 7.2.4 (Component 기반 UI 설계 및 REST API 연동)
- **Auth**: HttpSession (개발 편의성 및 세션 기반 보안 컨텍스트 구현)
- **Others**: JavaMail (비밀번호 재설정), Chart.js (통계 시각화)

3️⃣ 핵심 기능 및 설계 (아키텍처)

- **스터디 라이프사이클 관리 시스템 설계**
  - `Active` → `Arrived` → `Deleted` 상태 전이 모델 설계
  - 종료 시 (Arrived) 멤버 상태 변경, 모집 공고 마감, 신청서 취소를 하나의 트랜잭션으로 원자적 처리
- **권한 검증 레이어 분리**
  - 서비스 간 순환 참조 문제 해결을 위해 `StudyAccessService` 도입
  - 권한/조회 책임을 독립 계층으로 분리하여 의존성 단순화
- **REST API 명세 구조 정립**
  - `/api` 네임스페이스 분리로 SPA 라우팅 충돌 해결 및 프론트-백엔드 경계 명확화
- **ERD 설계 및 데이터 모델링**
  - User, Study, Recruitment, Application 도메인 관계 모델링 및 상태값 기반 정규화 수행

4️⃣ 기술적 문제 해결 (Troubleshooting)

- **순환 참조로 인한 서버 기동 오류**
  - **[실패]** 서비스 간 무분별한 상호 참조(`StudyService` ↔ `MemberService`)로 순환 의존성 발생 및 서버 기동 불가.
  - **[개선]** 공통 조회 및 검증 로직을 하위 레이어(`AccessService`)로 추출하여 의존성 단방향화.
- **SPA 라우팅과 API 경로 충돌**
  - **[실패]** 새로고침 시 브라우저가 직접 URL을 요청하면서 서버 컨트롤러와 경로가 겹쳐 404/405 오류 발생.
  - **[개선]** API 경로Prefix(`api/**`) 설정 및 미매칭 요청을 프론트엔드로 위임(Forward)하여 해결.
- **데이터 일관성 파괴 및 유실**
  - **[실패]** 스터디 종료 시 연관 도메인 데이터가 수동으로 관리되어 모집 공고나 신청서가 방치됨.
  - **[개선]** 일련의 상태 변화를 **하나의 트랜잭션 단위로 묶어 원자적 처리**함으로써 데이터 무결성 확보.

5️⃣ 결과 & 배운 점 (기술적 통찰)

- **도메인 정합성은 트랜잭션 경계 설계에서 결정된다**: 비즈니스 이벤트 단위로 데이터 상태 변화를 묶어야 시스템 일관성을 유지할 수 있음을 실전으로 경험했습니다.
- **의존성은 기능이 아니라 책임 기준으로 분리해야 한다**: 순환 참조 해결 과정에서 레이어 분리의 기준은 단지 코드를 나누는 것이 아니라 "책임의 범위"를 정하는 것임을 학습했습니다.
- **API 구조는 배포 이후 안정성에 직결된다**: SPA와 서버의 역할 경계를 명확히 구분하지 않으면 실제 운영 환경에서 예상치 못한 오류가 발생할 수 있음을 인지했습니다.

## 🎯 면접 Q&A

**Q1. MyBatis를 선택한 특별한 이유와 JPA와의 차이점?**
> **A.** 조인과 통계 쿼리에 대한 세밀한 통제권 때문입니다. MyBatis를 통해 쿼리 최적화와 가시성을 직접 확보하는 역량을 기르는 것을 최우선으로 두었습니다.

**Q2. 순환 참조 문제에서 AccessService 분리가 최선이었나요?**
> **A.** 비즈니스 로직과 읽기 전용 검증 로직이 섞여 있는 것이 근본 원인이었습니다. 이들을 분리함으로써 의존성 대상을 단일화했고, 결과적으로 코드의 가독성과 유지보수성이 비약적으로 향상되었습니다.

**Q3. 트랜잭션 관리 시 고려한 점은?**
> **A.** ACID 특성 보장과 성능 최적화입니다. 특히 복합 도메인 상태 변화 시 `@Transactional`을 통해 원자성을 보장하고, 조회 시에는 `readOnly = true`를 적용하여 불필요한 리소스 소모를 방지했습니다.
