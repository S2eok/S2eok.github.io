# Portfolio Master (합격권: 수치 & 행동 중심 버전)

## 1️⃣ 지원동기
**“정합성과 성능을 숫자로 증명하는 개발자”가 되겠습니다.**

StudyFleet에서 4개의 도메인(스터디, 멤버, 모집, 신청)이 엮인 종료 로직을 처리하며, 분산된 트랜잭션으로 인한 **데이터 유실 가능성 100%를 '0'으로 만드는 정합성 설계**를 주도했습니다. 단순 기능 구현이 아닌, `Active`에서 `Arrived`로의 상태 전이를 단일 원자적 트랜잭션으로 묶어 시스템의 신뢰도를 확보했습니다.

RE:FIT 프로젝트에서는 AI 연동 시 발생하는 **30초 이상의 응답 지연을 0.5초(submission 기준)로 98% 단축**한 경험이 있습니다. Tomcat 워커 스레드가 100%(200개 스레드 점유)에 도달하여 전체 서비스가 마비되는 상향을 목격하고, 이를 Redis 기반 비동기 Queue 아키텍처로 전환했습니다. 

단순히 비동기를 도입하는 것에 그치지 않고, `TransactionSynchronization`을 적용하여 **"DB 커밋 실패 시 불필요한 AI 호출이 발생하는 자원 낭비"를 원천 차단**했습니다. 이러한 수치 기반의 개선 경험과 예외 케이스까지 고려하는 꼼꼼함으로 귀사의 서비스 안정성을 책임지겠습니다.

## 2️⃣ 성장과정
**저는 "문제의 징후를 추적하여 구조를 재설계하는" 과정을 통해 성장했습니다.**

개발 초기, SPA 라우팅 충돌이나 순환 참조 오류를 단순한 '설정 오류'로 치부했을 때 비슷한 문제가 반복됨을 발견했습니다. 이때부터 에러 로그의 근본 원인을 아키텍처 레이어까지 추적하는 습관을 들였습니다.

- **로그 기반 병목 진단**: AI 서버 지연으로 인한 서버 중단 시, 스레드 덤프를 분석하여 **"Block 상태의 스레드 200개"**를 확인하고 비동기 전환의 근거로 팀에 제시했습니다.
- **의존성 그래프 시각화**: 서비스 간 순환 참조 발생 시, 직접 의존성 관계도를 그려 **`AccessService`라는 공통 조회 계층의 필요성**을 논리적으로 도출했습니다.
- **원칙 수립과 전파**: 팀 프로젝트에서 “DB 트랜잭션 선행 후 외부 메시지(WebSocket/Redis) 발행”이라는 정합성 원칙을 문서화하여 팀원들과 공유하고, 이를 인터셉터와 이벤트 리스너로 강제화했습니다.

이러한 **[데이터 기반 분석 - 구조적 대안 제시 - 팀 내 표준화]** 과정은 저의 가장 큰 개발적 자산입니다.

## 3️⃣ 성격의 장단점
### 장점: 끝까지 파고드는 구조화 능력
RE:FIT 프로젝트 당시, 한 팀원이 "구현이 간단한 `@Async`를 쓰자"고 제안했습니다. 하지만 저는 **"스레드 풀 관리의 어려움과 트랜잭션 전파 문제로 인한 데이터 유실 위험"**을 수치와 예시로 설명하며 Redis 큐 도입을 설득했습니다. "작동만 하는 코드"보다 "장애 상황에서도 데이터가 안전한 구조"를 선호하기 때문에 얻을 수 있었던 결과였습니다.

### 단점: 초기 설계 시의 깊은 고민
전체 시스템의 정합성을 고려하다 보니 초기 설계에 시간이 소요됩니다. 이를 보완하기 위해 **[1단계: 핵심 트랜잭션 흐름도 작성(1일) → 2단계: MVP 구현(3일) → 3단계: 리팩토링 및 예외 처리]**와 같이 타임라인을 명확히 구분하여 개발 속도와 품질의 균형을 유지하고 있습니다.

## 4️⃣ 협업 경험 (장면 중심)
**"팀의 기술적 부채를 해결하는 백엔드 리더"**

**[결정적 장면]**
팀 프로젝트 중, 프론트엔드 팀원으로부터 "채팅 연결 시 자꾸 인증 에러가 난다"는 리포트를 받았습니다. 확인 결과 WebSocket은 전역 HTTP Filter를 타지 않는다는 구조적 특성을 놓치고 있었습니다. 

저는 즉시 **공식 문서를 스터디하여 `ChannelInterceptor`의 존재를 확인**했고, 다음과 같이 3단계로 조치했습니다.
1.  **현상 분석**: 소켓 연결(Handshake)은 성공하나 첫 STOMP 프레임에서 인증 정보 부재 확인.
2.  **행동**: `CONNECT` 프레임 단계에서 JWT를 가로채 `SecurityContext`에 수동 주입하는 인터셉터 개발.
3.  **결과**: 보안 공백을 해소하고 채팅 연결 성공률 100% 달성.

이 과정을 통해 협업에서 **"안 되는 이유를 정확히 진단하고 대안을 제시하는 리더십"**의 중요성을 배웠습니다.

## 5️⃣ 기술적 어려움 & 극복 (면접 집중 대비)

### 1️⃣ [StudyFleet] 순환 참조와 데이터 정합성 (Action 중심)
- **상황**: `StudyService`와 `MemberService`가 서로를 참조하며 서버 기동 실패.
- **행동**:
    - ① 모든 서비스의 메서드를 **'상태 변경(CUD)'**과 **'권한/단순조회(R)'**로 분류.
    - ② 'R'에 해당하는 로직을 `StudyAccessService`로 통합 추출.
    - ③ 각 서비스가 AccessService만 참조하도록 의존성 단방향화(Refactoring).
- **결과**: 순환 참조를 100% 제거하고, 여러 서비스에서 중복되던 권한 체크 코드를 1곳으로 집중시켜 유지보수성 확보.

### 2️⃣ [RE:FIT] AI 지연 병목 현상 해결 (Metrics 중심)
- **상황**: 가상 피팅 요청 시 AI 서버의 30초 지연으로 인해 **API 타임아웃 발생 및 서버 멈춤 현상**.
- **행동**:
    - ① **Throttling 도입**: Redis List를 큐로 활용하여 초당 처리량을 Worker 스레드 수로 제한.
    - ② **비동기 정합성 확보**: `TransactionSynchronizationManager`를 통해 DB에 `PENDING` 상태를 먼저 확정한 후 큐에 발행.
    - ③ **응답 UX 개선**: 작업 Job ID를 即時 반환하여 유저가 다른 활동을 할 수 있도록 설계.
- **결과**: **최종 응답 시간 30,000ms → 500ms(정상 응답)로 개선**, 서버 스레드 점유율 대폭 감소.
