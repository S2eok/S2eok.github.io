# 프로젝트 정리: RE:FIT (AI 기반 패션 커뮤니티 & 중고 거래 플랫폼)

1️⃣ 전체 구조

① 프로젝트 한 줄 요약
**"AI 가상 피팅과 실시간 경매를 통해 중고 의류 거래의 ‘핏 실패’와 ‘가격 책정 부담’을 해결한 패션 거래 플랫폼"**

② 기획 배경 (문제 정의)
- **착용 불가**: 직접 입어볼 수 없는 중고 의류 특성상 사이즈/핏 실패가 잦음.
- **등록 포기**: 복잡한 사진 촬영 및 상세 정보 입력 과정으로 인한 상품 등록 이탈.

③ 해결 전략 (핵심 아이디어)
- **가상 피팅 시뮬레이션**: 생성형 AI(Gemini)를 활용한 아바타 착용 피드백 제공.
- **시장가 기반 실시간 경매**: WebSocket을 활용한 입찰 경쟁으로 가격 결정 부담 제거.
- **안전한 실시간 소통**: STOMP 기반 채팅과 JWT 인증을 결합한 신뢰 환경 구축.

2️⃣ 기술 스택 & 인프라

- **Framework**: Spring Boot 3.x, Java 17
- **Database**: MySQL, **Flyway** (팀 협업 스키마 이력 관리)
- **Cache & MQ**: **Redis** (비동기 작업 큐 및 캐싱)
- **Real-time**: WebSocket (STOMP), Spring Messaging
- **Auth**: Spring Security + JWT (Stateless 인증 및 소켓 보안 통합)
- **Infrastructure**: Nginx, Docker Compose

3️⃣ 핵심 기능 및 역할 (통합 버전)
- **백엔드 리드 및 비동기 아키텍처 설계를 전담**

- **비동기 가상 피팅 시스템 구축**
  - 생성형 AI 연동 시 발생하는 성능 병목을 해결하기 위해 **Redis 기반 비동기 Queue** 도입.
  - **Producer-Consumer 패턴** 적용으로 요청-처리를 분리하여 시스템 응답성 확보.
- **실시간 경매 및 채팅 인프라 설계**
  - WebSocket(STOMP)을 활용한 입찰/메시징 시스템의 전체 라이프사이클 구현.
  - 대화 기록 영속화 및 읽음 확인 로직 설계로 실무 수준의 통신 환경 구축.
- **보안 및 데이터 모델링 주도**
  - Spring Security와 WebSocket 인터셉터를 결합한 전역 인증 가드 구축.
  - Flyway 기반 마이그레이션 관리로 팀 내 데이터 정합성 유지.

4️⃣ 기술적 문제 해결 (Troubleshooting)

- **생성형 AI 지연(30s+)으로 인한 서버 자원 고갈**
  - **[실패]** 동기방식 호출 시 AI 서버 응답 대기로 인해 Tomcat 워커 스레드가 고갈되어 전체 시스템 마비.
  - **[개선]** 작업을 Redis 큐에 위임하고 즉시 응답을 반환하는 **비동기 구조** 도입. `TransactionSynchronization`을 통해 DB 커밋 후에만 작업을 위임하여 정합성 보장.
- **WebSocket 환경에서의 보안 공백**
  - **[실패]** WebSocket 통신이 Spring Security의 일반적인 HTTP 필터 체인을 타지 않아 인증 없이 접근 가능한 취급.
  - **[개선]** `ChannelInterceptor`를 구현하여 STOMP의 `CONNECT` 단계에서 JWT를 직접 검증하고 보안 컨텍스트를 주입하도록 고도화.
- **실시간 입찰 경쟁의 데이터 정합성**
  - **[실패]** 입찰 성공 메시지 전달(WebSocket)과 실제 최고가 갱신(DB) 간의 시차로 인한 정합성 이슈 우려.
  - **[개선]** **"DB 트랜잭션 선행 후 브로드캐스팅"** 원칙을 고수하여 WebSocket은 단순 전달 수단으로, 최종 정합성은 DB에서 보장하도록 설계.

5️⃣ 결과 & 배운 점 (기술적 통찰)

- **시스템 사고를 통한 병목 해결**: 단순 코드 최적화가 아닌 기술적 한계를 아키텍처 레벨(Redis Queue)에서 해결하는 경험을 얻었습니다.
- **트랜잭션과 비동기 통합의 중요성**: 비동기 작업을 다룰 때 DB 트랜잭션의 커밋 시점과 외부 큐 메시지 발행 시점을 일치시키는 정교한 설계의 중요성을 체감했습니다.
- **실시간 데이터 신뢰도 확보**: 분산 환경에서의 실시간 통신은 전달 속도보다 "마지막에 남는 데이터의 정합성"이 더 중요하다는 점을 학습했습니다.

## 🎯 면접 Q&A

### 🔥 핵심 기술 무기 (Architecture & Logic)

**Q1. 왜 Kafka가 아닌 Redis 큐를 선택했나요?**
> **A.** 프로젝트 규모와 리소스의 효율성 때문입니다. Kafka는 강력한 처리량을 제공하지만 대규모 클러스터 관리 오버헤드가 큽니다. 반면 Redis는 이미 캐시로 사용 중이었고, 간단한 List 연산만으로도 현 요구사항인 비동기 처리를 충분히 안정적으로 수행할 수 있다고 판단했습니다.

**Q2. TransactionSynchronization의 afterCommit()을 사용한 이유는?**
> **A.** 데이터 정합성 때문입니다. DB 커밋 전 작업을 큐에 넣으면, DB 장애로 저장에 실패해도 AI 작업은 실행되는 오류가 생깁니다. 이를 통해 "DB에 저장된 작업만 실행된다"는 원칙을 보장했습니다.

**Q3. WebSocket 인증을 필터가 아닌 인터셉터에서 처리한 이유는?**
> **A.** WebSocket은 최초 Handshake 이후 TCP 세션으로 유지되므로 일반적인 HTTP 필터 체인을 타지 않기 때문입니다. 따라서 메시지 프레임을 가로채는 인터셉터 단계에서 인증을 수행하는 것이 STOMP 구조에 가장 적합한 설계였습니다.

### ⚡ 실무 감식 (Security & Scalability)

**Q4. 트래픽 증가 시 아키텍처 확장 방안은?**
> **A.** 현재 내장된 메시지 브로커를 외부 Redis 브로커로 교체하여 다중 서버 간 메시지를 공유하고, AI 작업을 처리하는 Worker를 별도의 마이크로서비스로 분리하여 병목을 격리하는 Scale-out 전략을 취하겠습니다.

**Q5. 입찰 시 동시성 충돌은 어떻게 제어했나요?**
> **A.** 입찰 로직을 짧은 트랜잭션으로 묶어 DB 수준에서 영속성을 보장하고, 성공 건에 대해서만 웹소켓으로 전파합니다. 향후 더 높은 트래픽 환경에서는 Redis의 분산 락(Distributed Lock)을 도입하여 DB 부하를 선제적으로 제어할 계획입니다.
